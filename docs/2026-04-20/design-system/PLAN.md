# ErpGenex Enterprise Design System — خطة التنفيذ

**الإصدار:** 1.0  
**الإطار:** Frappe Desk + تطبيقات Omnexa  
**الهدف:** واجهة موحّدة على مستوى ERP عالمي (SAP / Oracle / Dynamics) مع **عدم كسر** منطق الخلفية أو سلوك DocTypes.

## المبدأ المركزي

كل التغييرات المرئية والهيكلية تمر عبر طبقتين:

| الطبقة | الوصف | الموقع الحالي |
|--------|--------|----------------|
| **A — Global shell** | رموز تصميم، شبكة، جداول، قوائم، رأس صفحة | `omnexa_core/public/css/omnexa_core.css` |
| **B — Adaptive behavior** | تمييز الحقول المطلوبة، مستويات الشبكة، إعادة التدفق عند التحميل الديناميكي | `omnexa_core/public/js/form_layout_optimizer.js` |

المرحلة **C** (اختيارية لكل DocType): إعادة ترتيب حقول محددة عبر Customize Form أو JSON في التطبيق — لا تُطبَّق دفعة واحدة على كل الشاشات.

## المراحل

### المرحلة 0 — الأساس (مكتملة / مستمرة)

- [x] حقن CSS عالمي عبر `app_include_css`
- [x] محسّن نماذج غير مدمر (`omnexa-form-optimized`)
- [ ] توثيق الرموز (tokens) ومزامنتها مع CSS — انظر `TOKENS.md`

### المرحلة 1 — المعاملات (Forms)

**النتيجة المستهدفة:** هيكل يقترب من المرجع (عنوان، حالة، إجراءات، كتل معلومات، جدول بنود، ملخص).

**في الكود العالمي (بدون لمس كل DocType):**

- شبكة قسم الحقول: حتى **3 أعمدة** على سطح المكتب، **2** على لوح، **1** على الجوال
- بطاقات أقسام بمسافات ثابتة، تسميات فوق الحقول
- جداول فرعية: رأس لاصق حيث يسمح DOM، صفوف متناوبة خفيفة، عرض كامل للجدول

**لكل نوع مستند حساس للإدخال السريع (Sales/Purchase Invoice، إلخ):**

- مراجعة ترتيب الأقسام في Customize Form ليطابق: **نواة المعاملة → التفاصيل المجمّعة → البنود → الملخص**
- طي الأقسام الثانوية افتراضياً حيث يناسب العمل

### المرحلة 2 — القوائم والتقارير

- قوائم Frappe: تلوين صفوف خفيف، اتساق ارتفاع الصف، عدم إخفاء الإجراءات الأساسية
- تقارير Query / Script: محاذاة عناوين وتباعد؛ عدم overrides كاسحة لـ DataTables دون اختبار

### المرحلة 3 — شريط إجراءات سفلي (Sticky footer bar)

**Frappe يوفّر بالفعل `page-head` لاصقاً مع أزرار الحفظ/الإرسال.**  
إن لزم **شريط سفلي مكرر** (مطابق للمرجع البصري): يتطلّب مكوّن JS مخصّص يستنسخ أو يعكس شريط الإجراءات — **مرحلة لاحقة** بعد اختبار على 2–3 DocTypes رئيسيين لتجنب تضارب التركيز وقارئ الشاشة.

## معايير النجاح (قابلة للقياس)

1. أي نموذج معاملة: معلومات الترحيل والطرف والشركة ضمن أول قسم أو ثانٍ في التمرير على 1920×1080
2. لا أكثر من **3 حقول غير جدول** في الصف الواحد على سطح المكتب (ما عدا الاستثناءات المعرّفة في DocType)
3. جدول البنود: الرؤية الكاملة للعرض، رأس الجدول يبقى ظاهراً أثناء التمرير داخل منطقة الجدول حيث يدعم المتصفح
4. تباعد من `TOKENS.md` فقط (4 / 8 / 16 / 24 / 32 px)

## تعارض محتمل وحله

| المخاطرة | التخفيف |
|-----------|---------|
| قواعد شبكة قوية تكسر Tab Break في نماذج معينة | استثناء الحقول من نوع Tab/Section Break من الشبكة (موجود في CSS) |
| sticky على رأس الجدول الفرعي داخل حاوية `overflow` | اختبار على DocTypes ذات بنود طويلة؛ تعديل انتقائي لاحقاً |
| ازدواج أزرار الحفظ (إن أُضيف شريط سفلي) | توحيد المصدر أو إخفاء النسخة الثانوية عند الطي |
